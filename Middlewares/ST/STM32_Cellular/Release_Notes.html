Purpose

Overview (RTOS agnostic) of the X-CUBE-CELLULAR (Core of the STM32 Cellular software expansion for STM32Cube)

Overview of the X-CUBE-CELLULAR used in Azure RTOS environment.

Overview of the X-CUBE-CELLULAR used in FreeRTOS environment.

  • The Cellular Middleware is the SW stack that allows to drive cellular modems from STM32 MCUs.
  • User can choose between IP stack in STM32 (NetX Duo in Azure RTOS or LwIP with FreeRTOS, called STM32 IP) or in modem (called Modem IP mode) thanks to a compiler switch.
  • Upper layers APIs are composed of COM (data plane) APIs and Control (Control plane) APIs.
  • AT is an AT command handler, it is splitted in 2 parts, core (modem agnostic part, generic) and custom (modem dedicated part, one per modem).
  • PPPosIF is the interface for the IP stack (Netx Duo or LwIP) for the STM32-Modem serial interface (through IPC, that abstracts the UART)
  • CST/CS is Cellular Service (Task and library) this is the Cellular dynamic part.
  • Reminder: The Cellular middleware is compliant with MISRA-C®:2012 with some minor limitations and checked against a static analysis tool to eliminate run-time errors. Reports are available on demand under NDA.
  • There is at least one application to use Cellular middleware and it is available for all supported setups.

Maintenance release and product update

This V7.0.0 is a big change regarding previous versions

  • Azure RTOS (ThreadX and NextX Duo) has been added to FreeRTOS/LwIP
  • FreeRTOS/LwIP is available through a dedicated add-on (not included in the main delivery)
  • It uses RTOSAL as before but a new cmsis_os2 has been added, to support ThreadX, to the cmsis_os2 used (for FreeRTOS) in previous deliveries.

Other improvements

  • cst targetstate commands renamed into cst modem
  • APN automatic selection uses ICCID to support SIM card with Multi-IMSI features
  • cst info command, new information is displayed (for debug when traces are deactivated)
  • Code refactoring for Cellular Service API (API between CS Task (CST) and AT-Core)
    • The code dedicated to Socket Mode (IP stack in modem) as been put in a separate file: Cellular_Service_Data
  • Code refactoring for AT Core
    • The file at_modem_signalling has been split in 2 parts: at_signalling_build + at_signalling_analyze
    • The file sysctrl has been renamed at_sysctrl
    • Code dedicated to Socket Mode (IP stack in modem) has been under compilation flag to minimize code footprint if IP stack is STM32 side.
  • Code refactoring for AT custom modem
    • The file at_modem_custom_specifc has been split in 2 parts: new file created at_modem custom_sid. Previously, all Service ID (SID) coming from Cellular Service were managed in a big switch/case in the file at_modem_custom_specifc. Now, all these SID are managed in dedicated functions called thanks a Look-up table implemented in the file at_modem_custom_sid. The entry point is a new functions called at_custom_SID_XXX (where XXX is the modem name). This modifcation is purely ‘cosmetic’ to keep reasonnable file size, and as a consequence, if you have a custom modem driver, this split is not mandatory but highly recommended.
    • The file sysctrl_specific has been renamed at_sysctrl_specific
    • Some structures using pointers has been prefixed by “p_”
    • Some internal function has been renamed
    • Code dedicated to Socket Mode (IP stack in modem) has been under compilation flag to minimize code footprint if IP stack is STM32 side. The file at_custom_modem_socket is required now only if using Modem IP stack.

Bug fixes

  • Missing PS network domain + control plane detach. New API function added: CS_deregister_net().
  • AT : [Azure] optimization of Socket Data Receive when polling requests
  • AT : XCC crashes (null pointer check)
  • AT : Serial Number and IMEI are incorrect with modem Monarch
  • AT : (T1SC only) X-CUBE-CELLULAR - 6.01 - Case cellular modem ip sockets - issue
  • AT : (Sequans only) A tempo has been added after modem power off (to avoid problem if modem is restarted immediately).
  • IPC : Azure NetXDuo (IP stack in STM32) where “Modem UART RX interrupt rearm” was failing

Known Limitations

  • Low Power concerns only modem part (STM32 low power is not yet implemented).
  • No modem low power in IP stack in STM32 mode (i.e. only in modem IP mode).
  • No vertical test application to test low power is run (only low level tests on FreeRTOS run)
  • COM : shutdown function not supported.
  • COM : multi-threads using same socket not supported, for example one thread is wait for receiving data on a socket, another thread cannot send data on the same socket
  • The COM API functions com_send, com_sendto or com_recv, com_recvfrom may block even if application has defined MSG_DONTWAIT. It is due to the nature of AT command/response exchange with Modem.
  • COM : UDP in not-connected mode (normal UDP mode) is not yet supported by Sequans GM01Q modem FW (sendto and recvfrom not supported).
  • Ctrl : Deinit and Stop of platform is not supported
  • COM : The error code returned by the Modem is not reported to the application via socket BSD.
  • CellularIoT samples is an application example demonstrating the use of display and sensors of B-L462E-CELL1 board. First line displayed should be the UTC date and time. When data connection is established, with the actual application, Date/Time is requested one time only to a distant server and sometimes the server doesn’t respond to this request; in this case date/time isn’t displayed and treatment continue.

Main Changes

This V6.0.0 adds the following features:

  • Modem low power supported for BG96 and T1SC (By defaut low power mode is not activated)
  • APN automatic selection
  • Automatic SIM selection
  • LTE Bands & Radio Access Technology selection
  • Boot menu has been removed but command line is maintained however users can easily remove it
  • Cellular middleware init and start is now done by application (no more from main)
  • SIM refresh sent to application
  • After registration, application is informed of PSM states changes
  • Cellular middleware version available for application
  • New API for control is added. Data Cache is still used internally but no more an API for applications, it is now covered by Cellular API library.
  • HW setups supported changed

Bug fixes

  • Data cache callback
  • SIM state during modem initialization
  • IPC errors
  • Time / Date corrections
  • MDK warnings correction
  • MISRA, CodeSonar & CodeReview corrections

Known Limitations

  • P-L496G-CELL01 (UG96) is no more supported
  • No modem low power in LwIP mode (i.e. only in modem socket mode).
  • for STWINKT1B only modem socket mode is delivered (i.e. LwIP not delivered).
  • LwIP suspend/resume not recommended as introduce some limitations.
  • The COM API functions com_send, com_sendto or com_recv, com_recvfrom may block even if application has defined MSG_DONTWAIT. It is due to the nature of AT command/response exchange with Modem.
  • UDP in not-connected mode (normal UDP mode) is not yet supported by Sequans GM01Q modem FW (sendto and recvfrom not supported).
  • For STM32CubeIDE in Lwip configuration, Warnings in ThirdParty Lwip (pppos.h).
  • For MDK_ARM in LwIP, warning in Middlewares Thirdparty LwIP.
  • APIs for some power commands are missing.
  • COM : The error code returned by the Modem is not reported to the application via socket BSD.
  • COM : shutdown function not supported.
  • L462 full BSP (QSPIs drivers and examples are missing).
  • One socket managed by more than one thread.
  • COM_ICC (to access ST4SIM through NDLC middleware) is not officially supported and is not supporting low power(It will in next release).
  • Few MISRA-C®:2012 rules (will be corrected in next delivery)
